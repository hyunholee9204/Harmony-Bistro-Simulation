# 🍽 Harmony Bistro Simulation

## 📌 프로젝트 소개
**Harmony Bistro Simulation**은 레스토랑 웹 페이지를 직접 구현하고,  
그 위에 사용자 행동에 따라 시스템이 반응하는 인터랙티브 요소를 결합한 프로젝트입니다.
 
손님 입장 → 주문 → 주방 조리 → 완료 → 결제까지 이어지는 흐름을  
상태 기반 로직으로 설계하여 실제 레스토랑 운영 과정을 시뮬레이션 형태로 구현했습니다.

특히 사용자 클릭과 같은 입력에 따라 테이블 상태, 주문 진행 상황, 결제 가능 여부가 동적으로
변화하도록 구성하며, **웹 UI와 로직이 유기적으로 연결된 인터랙티브 시스템**을 구현하는 데 초점을 맞췄습니다.

---

## 🎯 프로젝트 목표

- 단순 UI 구현이 아닌 **상태 기반 로직 중심의 웹 애플리케이션 개발**
- 사용자 행동에 따라 동작하는 **인터랙티브 시스템 구현**
- 실제 서비스 흐름을 반영한 문제 해결 경험

## 웹 페이지

### 왜 Harmony Bistro 인지?

"Harmony Bistro"라는 이름은 다양한 음식이 조화를 이루는 레스토랑이라는 의미를 담고 있습니다.

**한식, 중식, 양식** 등 서로 다른 카테고리의 요리를 한 공간에서 제공하는 레스토랑을 주제로 기획되었습니다.
식사를 할 때 음식의 종류가 다양할수록 선택의 폭이 넓어지고, 그에 따라 만족도 또한 높아진다고 생각합니다.

이러한 아이디어를 바탕으로, 각기 다른 메뉴와 시스템 요소들이 하나의 흐름 안에서
자연스럽게 연결되고 조화를 이루는 모습을 표현하고자 "Harmony"라는 단어를 사용하게 되었습니다.

## 🎮 주요 기능

### 👥 테이블 및 손님 관리
- 랜덤 인원의 손님 입장 및 대기열(Queue) 관리
- 선입선출(FIFO) 방식의 테이블 배정 시스템
- 테이블 상태 기반 관리  
  `EMPTY → OCCUPIED → COOKING → READY`

---

### 📝 주문 시스템
- 한식 / 중식 / 양식 카테고리 기반 랜덤 메뉴 주문 생성
- 테이블 인원 수에 따라 주문 수량 동적 결정
- 주문 데이터와 가격 정보를 객체 형태로 관리
- 영수증 UI를 통한 주문 내역 확인

---

### 👨‍🍳 주방 조리 시스템
- 한식 / 중식 / 양식 주방 분리 구조
- 주문 메뉴가 카테고리에 따라 해당 주방으로 자동 전달
- 메뉴 단위 조리 완료 버튼 제공
- 실제 조리 흐름을 반영한 주방 처리 구조 구현

---

### ⏱ 조리 완료 및 상태 관리 로직
- 메뉴 단위 완료 처리 및 servedCount 기반 진행 관리
- 모든 메뉴가 완료된 경우에만 테이블 상태를 READY로 변경
- 부분 완료 상태에서는 결제 불가능하도록 제한
- 상태 기반으로 UI 및 기능 동작 제어

---

### 💳 결제 시스템
- 계산대에서 테이블별 주문 내역 및 금액 확인
- 음식 가격 + 팁을 포함한 총 금액 계산
- 모든 음식 완료 시에만 결제 가능
- 결제 완료 시 테이블 상태 초기화 및 매출 반영

---

### 🧾 영수증 UI
- 주문 메뉴별 가격 표시
- 총 금액 자동 계산
- 실제 영수증 형태를 반영한 UI 구성

---

### 🎯 인터랙티브 시스템
- 사용자 행동(클릭)에 따라 상태와 UI가 동적으로 변화
- 테이블 상태에 따라 제공되는 기능이 달라지는 구조
- 주방 → 테이블 → 계산대로 이어지는 흐름을 인터랙티브하게 구현
---

## 🧠 시스템 설계 포인트

### ✔ 상태 기반 설계 (State Management)

```
EMPTY → OCCUPIED → COOKING → READY → EMPTY
```

- 테이블 상태를 기준으로 전체 로직을 제어
- 상태에 따라 가능한 기능을 제한하여 비정상적인 흐름 방지
- 상태 변화에 따라 UI가 동적으로 변경되도록 설계

---

### ✔ 진행 상태 추적 로직 (servedCount)

- 주문 메뉴 수와 완료된 메뉴 수를 비교하여 진행 상태 관리
- `servedCount`를 활용하여 모든 메뉴 완료 여부 판단
- 전체 완료 시에만 결제가 가능하도록 제한

---

### ✔ 이벤트 기반 인터랙션 구조

- 사용자 클릭 이벤트 중심으로 시스템 동작 설계
- 행동 → 상태 변화 → UI 반영의 흐름 구조
- 동일한 UI에서도 상태에 따라 다른 기능 제공

---

### ✔ 기능 단위 분리 (Separation of Concerns)

- 주문 / 주방 / 결제 로직을 기능별로 분리
- 각 기능이 독립적으로 동작하도록 설계
- 유지보수성과 확장성을 고려한 구조
